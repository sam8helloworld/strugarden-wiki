# システムアーキテクチャ

対人戦を成立させるための最小構成のシステムアーキテクチャを記述する。ライブラリ/フレームワーク選定には踏み込まず、**責務分割と通信パターン** に焦点を当てる。

## 2. コンポーネント責務

### 2.1 クライアント

- UIレンダリング・プレイヤー入力取得
- 戦闘シミュレータを内蔵し、**サーバ同期前のローカル予測**を行う (ターン制ゆえ入力予測の重要度は低め)
- マスターデータ (ジョブ/スキル/マップ定義) はクライアントに配布され、UI表示や事前バリデーションに使用
- 表示のための**補間アニメーション**は UI層のみが担当。シミュレータは整数座標/整数状態で進行する

### 2.2 API Gateway (REST/RPC)

以下の非戦闘的な操作を担当:

- 認証・認可
- キャラクター作成・編集
- デッキ編集
- 装備変更
- マッチ履歴・リプレイ取得
- ランキング取得

詳細エンドポイントは [`api.md`](./api.md)。

### 2.3 Battle Realtime Gateway

- クライアントとの双方向通信 (WebSocket または gRPC streaming を想定)
- `Battle Session` との間でメッセージをルーティング
- プレイヤー切断時の再接続サポート
- `フェイズタイマー` の通知 (サーバ側タイマーが権威)

### 2.4 Battle Session

- 1マッチごとに1インスタンス生成される状態機械
- **権威的 (authoritative)** な戦闘ステートを保持
- `BattleEngine` (決定論シミュレータ) を実行し、入力を受けてステートを進める
- 試合終了時に以下を永続化:
  - `MatchRecord`
  - `Replay` (シード + イベント列)
- 試合並列実行に対応 (1 プロセスあたり N セッション、またはセッション単位に worker をフォーク)

### 2.5 Matchmaker

- 闘技場モード向け。キューを管理し、条件が揃ったらマッチを組成。
- `MMR` 差、待機時間、デッキ制限等を加味してマッチング。
- マッチ確定後 `Battle Session` を生成し、全員に通知。

### 2.6 Job Worker (バッチ/スケジュールド)

- **日次**: 資源拠点争奪戦の功績ランクリセット (毎日06:00)
- **週次**: 拠点支配権更新 (水曜メンテ相当)、シーズンランキングスナップショット
- **任意**: 不正検出 (連敗ブースト、放置試合、ペアリング分析)

## 3. 戦闘通信パターン

戦闘はターン制のため、FPSのような60Hzの状態同期は不要。以下のパターンで十分:

```
[各フェイズ]
  サーバ → 全員:  PhaseStart(phase, deadline_ts)
  クライアント    入力収集 (UI操作)
  クライアント → サーバ:  SubmitInput(unit_id, command)  (複数回送信可)
  クライアント → サーバ:  ConfirmPhaseInput()            (入力確定)
  (全員確定 or デッドライン経過で)
  サーバ: エンジン実行
  サーバ → 全員:  PhaseResolved(events[])
```

- **他プレイヤーのプレビュー入力は送らない** (入力の秘匿性 — バッティング読み合いが競技性の根幹)
- `SubmitInput` は反映前なら何度でも上書き可能
- デッドライン経過で未確定入力は `wait` として扱う

## 4. 決定論的シミュレータの位置付け

- サーバ/クライアントで **同じシミュレータバイナリ** (または同じ言語で書かれた同じコードベース) を共有
- 入力 `ActionCommand` 列 + PRNG シード から唯一の結果を計算できる
- 利用用途:
  - サーバ: 権威実行
  - クライアント: プレビュー・観戦・リプレイ再生
  - AI (闘錬の間): 仮想プレイヤーとして入力を生成
- 実装言語は問わないが、サーバとクライアントで実装が分かれるなら **コンフォーマンステスト** (シードつきの入力列を両者に流し結果一致を確認) を CI で常時回す

## 5. 永続化

RDBMS は以下を保存:

- `Account`, `Character`, `Inventory`, `CharacterDeck`, `Familiar`, `FamiliarContract`
- `MatchRecord`, `MatchParticipant`
- `PlayerArenaStats`, `GuildArenaStats`, `PlayerConquestStats`, `GuildConquestStats`
- `TrainingHallProgress`
- `BasePoint`, `ConquestSeason`, `ArenaSeason`

BLOBストア (オブジェクトストレージ相当) は:

- `Replay` (バイナリ or JSON。将来圧縮・削除しやすいよう別ストア)

インメモリ/揮発ストア (KVS想定) は:

- `BattleSession` (アクティブなセッションとその揮発状態)
- `MatchmakerQueue` (キューイング情報)
- `RealtimeConnection` (接続ハンドル)

## 6. 信頼性・再接続

- **セッション状態の冗長性**: 短時間の障害に耐えるため、KVSへの定期スナップショット (1ターンに1回等) を取る。
- **再接続**: WebSocket断線時、同一 `session_id + unit_id` で再接続すれば続きから再開。未送信ステートは最新スナップショットから再送。
- **切断放置対策**: 連続3フェイズ入力なしで自動投了 (モード依存)。

## 7. 不正対策 (最低限)

- **入力検証**: クライアントから来るコマンドは全部サーバ側で再検証 (移動経路の妥当性、射程、APの消費可否等)。
- **クライアント予測乖離検出**: プレビュー結果とサーバ確定結果が違えばクライアントはサーバ側を信じる。
- **リプレイ検証**: 試合直後にサーバがリプレイを再生して勝敗が一致することを確認 (自己整合性チェック)。
- **Rate Limiting**: マッチキュー連投、キャンセル連発を制限。

## 8. スケーラビリティ観点

- **戦闘セッション**はプロセスをまたがない設計 (セッションごとに single-writer)。水平スケールはセッション数で。
- **マッチメーカー**は在庫キューを Redis 等に保持しつつ、複数プロセスで再配分。
- **永続化**は読み取り比率が高い (ランキング、リプレイ閲覧) ためキャッシュ戦略を設計する。

## 9. ログ・テレメトリ

- 戦闘開始/終了、マッチメーキング成立、投了、切断 は構造化ログ必須。
- スキル発動回数、状態異常付与回数、勝率はメトリクス化してダッシュボード化 (バランス調整用)。
- クラッシュレポート (サーバ/クライアント両方) を集約。

## 10. 開発環境

- マスターデータリポジトリは **ソースコードとは別のフォルダ** (運用が編集する想定)。
- バランス調整は YAML の差分で出せるように。
- シミュレータのユニットテストは **シード + 入力列 + 期待結果** のスナップショットテストが主力。
